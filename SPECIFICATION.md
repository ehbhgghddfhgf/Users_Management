# 🔧 Техническое задание

## 📌 Цель задания
Разработать модуль **«Управление пользователями»** на Vue 3 + TypeScript. Задание направлено на оценку архитектурного мышления, чистоты кода, масштабируемости и зрелости технических решений.

## 📈 Контекст и ожидания
Представьте, что приложение будет активно развиваться и поддерживаться в долгосрочной перспективе. Ожидается, что решения в коде будут рассчитаны на:
* Рост команды.
* Усложнение бизнес-логики.
* Добавление нового функционала без полной переработки архитектуры.

> **Важно:** Нам не важна идеальная пиксельная верстка. Нам важна логика компонентов, продуманная архитектура и чистота реализации composables / stores. Верстка может быть минималистичной — главное инженерный подход.

## 🧩 Функциональные требования

### 1. Страницы (vue-router)
* `/users` — список пользователей.
* `/users/new` — создание пользователя.
* `/users/:id/edit` — редактирование пользователя.

### 2. Список пользователей
**Запрос:** `GET /api/users?page=1&limit=40&search=...`

**Таблица должна содержать:**
* ID, ФИО, Email.
* Дата регистрации.
* Статус (Активен / Заблокирован).
* Действия (Кнопка «Редактировать»).

**Функциональность:**
* Серверная пагинация.
* Поиск по имени/email.
* Сохранение текущей страницы и фильтрации в URL (чтобы при обновлении страницы фильтры не сбрасывались).
* **⚠️ Важно:** пагинация и фильтрация должны выполняться на стороне сервера.

### 3. Создание и редактирование пользователя
**Методы API:** `POST /api/users` и `PATCH /api/users/:id`

**Поля:**
* ФИО, Email, Статус.

**Требования к валидации:**
* Проверка обязательности полей.
* Проверка корректности email через regex.
* Архитектура валидации должна быть масштабируемой (возможность легко добавлять новые правила). 
* **Рекомендуется** реализовать через composable вида `useValidate`.

## 🛠 Технические ограничения

### ✅ Разрешено:
* **Core:** Vue 3 (Composition API) + TypeScript.
* **State:** Pinia.
* **Routing:** vue-router.
* **HTTP:** axios.
* **Styles:** SCSS (строго методология **BEM**).
* **Server:** json-server (или любой mock-сервер).

### 🚫 Запрещено:
* UI-библиотеки (Vuetify, Element UI, Ant Design и др.).
* Tailwind CSS / Bootstrap.
* CSS-in-JS.
* Любые сторонние библиотеки для UI-компонентов.

## 🔍 Критерии оценки (Архитектура)
Нам особенно важно понять, как вы проектируете систему. Обратите внимание на:
1.  **Разделение ответственности:** Четкое разграничение страниц, переиспользуемых UI-компонентов, бизнес-логики и работы с API.
2.  **Изоляция API:** Логика запросов и трансформации данных должна быть вынесена из компонентов.
3.  **Масштабируемость:** Насколько легко будет добавить новую сущность (например, «Роли») в вашу текущую структуру.
4.  **Чистота кода:** Именование переменных, типизация в TS, отсутствие "магических чисел".
